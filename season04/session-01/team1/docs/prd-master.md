**배경:**

**발견된 핵심 문제:**
> "파업이 없으면 앱을 쓸 이유가 없다"

- 파업은 연간 몇 회 발생 → 나머지 기간 앱 사용 이유 부재
- 리텐션 제로 → 파업 때마다 재설치 필요
- 수익 모델이 파업 발생에 의존적 → 비즈니스 불안정
- 카풀/대절의 네트워크 효과 부재 → 유저 없으면 서비스 무용

**결론:**
"파업 출근 비서"에서 "**출퇴근 비서**"로 정체성 확장 필요. 파업 대응은 핵심 기능으로 유지하되, 상시 가치를 제공하여 일상적 사용을 유도해야 함.

## 1. 문제 정의

### 1.1 문제 상황

서울/경기 지역의 버스 파업이 발생하면 출퇴근자들은 다음과 같은 어려움을 겪는다:

- **정보 분산**: 파업 정보가 서울시/경기도청 보도자료, 지자체 SNS, 문자 안내, 뉴스 기사 등 여러 채널에 흩어져 있음
- **영향도 파악 어려움**: 파업 소식은 뉴스로 접하지만, 내가 타는 노선이 영향받는지 즉시 알기 어려움
- **대체 경로 탐색 시간 소모**: 대체 교통편을 직접 검색하고 비교해야 함
- **의사결정 어려움**: 출발 시간을 얼마나 앞당겨야 하는지 판단 어려움
- **타이밍 문제**: 파업 당일 새벽, 혼란 속에서 빠른 의사결정 필요

### 1.2 현재 대안의 한계

| 현재 대안 | 제공하는 것 | 부족한 점 |
|-----------|-------------|-----------|
| 뉴스/포털 | 파업 발생 사실 전달 | 내 노선 영향도, 대체 경로 없음 |
| 네이버/카카오 지도 | 일반 경로 탐색 | 파업 상황 미반영, 실시간 업데이트 없음 |
| 교통 정보 앱 | 실시간 도착 정보 | 파업 시 운행 중단 정보 부정확 |
| 서울시/경기도 공식 채널 | 공식 발표 | 개인 맞춤 정보 없음, 접근성 낮음 |

### 1.3 핵심 인사이트

> *"파업 알림"은 이미 뉴스가 해준다. 아무도 안 해주는 건 **"그래서 나는 어떻게 출근해?"**이다.*

현재 파업 정보를 **통합·요약·개인화하여 전달하는 서비스는 부재**함.

## 2. 타겟 유저

### 2.1 1차 타겟: 경기도 광역버스 출퇴근러

가장 타격이 큰 유저 그룹. 광역버스 의존도가 높고, 대체 교통편 접근성이 낮음.

### 2.2 페르소나

| 항목 | 내용 |
|------|------|
| 이름 | 김직장 (32세, 판교 IT기업 개발자) |
| 거주지 | 수원시 영통구 |
| 출근 루트 | 영통역 → 3100번 광역버스 → 판교역 (약 1시간) |
| Pain Point | 파업 뉴스 봤는데 3100번이 해당되는지 모르겠음. 지하철로 가면 1시간 40분인데, 몇 시에 나가야 하지? |
| 니즈 | 전날 밤에 알아서 새벽 지하철 탈 준비, 대체 경로와 소요시간 비교 |

### 2.3 사용자 세그먼트별 니즈

| 세그먼트 | 니즈 | 특징 |
|---------|------|------|
| **직장인** | 출근 전 사전 대응 (전날 밤 ~ 당일 오전 6시) | 시간 절약이 핵심 |
| **학생** | 등교 시간 유연성 낮음 (빠른 알림 필수) | 즉시성 중요 |
| **시니어** | 명확한 안내 필요 | 대체 수단 활용 어려움 |
| **회사 총무/HR** | 직원 출근 대응 공지 필요 | B2B 확장 가능성 |

### 2.4 확장 타겟

- 서울 시내버스 이용자: 대체 노선이 많아 영향도 확인만 필요
- 지하철 이용자: 지하철 파업 시 확장 대응

## 3. 핵심 가치 제안

### 3.1 한 문장 정의

> **"파업 떴을 때, 30초 만에 나만의 대체 출근 루트를 알려준다"**

### 3.2 제품 비전

서울/경기 버스 파업 발생 시,
**사용자가 아무것도 검색하지 않아도 → 푸시 알림과 요약 페이지를 통해 → "어디에서 어떤 버스가 영향을 받는지"와 "대신 어떻게 이동할 수 있는지"를 즉시 알 수 있도록 한다.**

### 3.3 핵심 가치

| 가치 | 설명 |
|------|------|
| **즉시성** | 파업 정보 확인부터 대체 경로 확인까지 30초 이내 |
| **개인화** | 내 출퇴근 경로 기반 맞춤 정보 |
| **실행 가능성** | 단순 정보가 아닌, 바로 행동할 수 있는 가이드 |
| **통합** | 분산된 파업 정보를 한 곳에서 확인 |

## 4. 기능 정의

### 4.1 핵심 목표

우리 앱은 **2가지 핵심 목표**를 중심으로 설계한다:

| 목표 | 설명 | 핵심 질문 |
|------|------|-----------|
| **목표 1: 신뢰도 높은 알림** | 파업 정보를 충실하고 정확하게 전달 | "파업이 언제, 어디서, 어떤 노선에 영향을 주는가?" |
| **목표 2: 대체 대안 제공** | 파업 시 실행 가능한 출퇴근 대안 제시 | "그래서 나는 어떻게 출근해?" |

```
┌─────────────────────────────────────────────────────────┐
│                    파업 출근 비서                         │
├────────────────────────┬────────────────────────────────┤
│   목표 1: 신뢰도 높은 알림  │     목표 2: 대체 대안 제공      │
├────────────────────────┼────────────────────────────────┤
│ • 파업 정보 수집/검증      │ • 출퇴근 경로 등록              │
│ • 파업 상태 관리          │ • 내 노선 영향도 체크           │
│ • 푸시 알림 발송          │ • 대체 경로 추천               │
│ • 파업 상세 페이지        │ • 소요시간 비교                │
└────────────────────────┴────────────────────────────────┘
```

---

### 4.2 목표 1: 신뢰도 높은 알림

파업 정보를 **정확하고 신속하게** 전달하는 것이 첫 번째 목표다.

#### 4.2.1 기능 목록

| # | 기능 | 설명 | 우선순위 |
|---|------|------|----------|
| 1 | 파업 정보 수집 | 공식 채널에서 파업 정보 수집 및 검증 | P0 |
| 2 | 파업 상태 관리 | 예고/확정/진행중/종료 상태 구분 | P0 |
| 3 | 푸시 알림 발송 | 파업 확정 시 사용자에게 즉시 알림 | P0 |
| 4 | 파업 상세 페이지 | 영향 지역, 노선, 시간대 상세 안내 | P0 |
| 5 | 알림 개인화 | 내 지역/노선 기반 맞춤 알림 | P1 |

#### 4.2.2 파업 상태 정의

| 상태 | 설명 | 알림 발송 |
|------|------|-----------|
| **파업 예고** | 파업 가능성 공식 발표 (협상 중) | X |
| **파업 확정** | 파업 실시 확정 (협상 결렬) | O |
| **부분 파업** | 일부 노선/시간대만 운행 중단 | O |
| **파업 종료** | 정상 운행 재개 | O |
| **파업 취소** | 예고 후 취소됨 | O (예고 수신자만) |

#### 4.2.3 신뢰도 확보 원칙

- **1차 출처 우선**: 서울시/경기도청 공식 발표 > 버스조합 > 언론 보도
- **교차 검증**: 복수 출처 확인 후 알림 발송
- **미확인 정보 배제**: SNS 루머, 추측성 정보는 절대 반영 안 함
- **출처 명시**: 모든 정보에 출처와 기준 시각 표시

### 4.3 목표 2: 대체 대안 제공

파업으로 영향받는 사용자에게 **실행 가능한 대안**을 제시하는 것이 두 번째 목표다.

#### 4.3.1 대체 대안 유형

파업 시 제공할 수 있는 대안은 **4가지**로 분류한다:

| # | 대안 유형 | 설명 | 대상 | MVP |
|---|-----------|------|------|-----|
| 1 | **최적 경로 (대중교통)** | 파업 안 하는 교통수단으로 경로 재탐색 | 개인 | O |
| 2 | **택시** | 개인 택시 호출 | 개인 | X |
| 3 | **카풀** | 같은 경로 유저 매칭, 비용 분담 | 개인/커뮤니티 | X |
| 4 | **버스 대절** | 회사/단체 단위 전세버스 | B2B | X |

```
파업 발생 시 대안 흐름:

버스 파업 → 지하철로 최적 경로 재탐색
지하철 파업 → 버스로 최적 경로 재탐색
전면 파업 → 택시/카풀/버스 대절 안내
```

#### 4.3.2 MVP: 최적 경로 (대중교통)

MVP에서는 **파업하지 않는 대중교통을 활용한 최적 경로**에 집중한다.

| # | 기능 | 설명 | 우선순위 |
|---|------|------|----------|
| 1 | 출퇴근 경로 등록 | 출발지/도착지 사전 등록 | P0 |
| 2 | 내 노선 영향도 체크 | 등록 경로가 파업 영향권인지 표시 | P0 |
| 3 | 대체 경로 추천 | 파업 안 하는 교통수단 기반 최적 경로 제시 | P0 |
| 4 | 소요시간 비교 | 평소 vs 대체 경로 시간 비교 | P1 |
| 5 | 권장 출발 시각 | 대체 경로 기준 출발 시간 안내 | P1 |
| 6 | 지도 앱 연결 | 네이버/카카오 지도로 바로 이동 | P2 |

**최적 경로 제공 범위:**
- 인근 지하철 노선 및 역 정보
- 파업 영향 없는 다른 버스 노선
- 환승 필요 시 간단한 이동 요약
- 도보 가능 거리 안내

#### 4.3.3 Phase 2: 택시

| 기능 | 설명 |
|------|------|
| 택시 호출 연동 | 카카오택시/타다 등 연결 |
| 예상 요금 안내 | 출발지-도착지 기준 요금 표시 |
| 할증 정보 | 파업 당일 수요 증가로 인한 할증 안내 |

#### 4.3.4 Phase 2: 카풀

| 기능 | 설명 |
|------|------|
| 카풀 매칭 | 같은 경로 유저 연결 |
| 비용 분담 | 유류비/톨비 자동 계산 |
| 채팅 | 카풀 참여자 간 소통 |

#### 4.3.5 Phase 3: 버스 대절 (B2B)

| 기능 | 설명 |
|------|------|
| 회사 단위 수요 집계 | 같은 회사 직원 출퇴근 루트 분석 |
| 전세버스 매칭 | 버스 업체 연결 |
| HR 대시보드 | 직원 출퇴근 현황 관리 |

### 4.4 MVP vs 향후 확장

#### MVP (1차 출시)

| 목표 | MVP 범위 |
|------|----------|
| 목표 1: 알림 | 파업 확정 알림 + 상세 페이지 (수동 정보 입력) |
| 목표 2: 대안 | **최적 경로 (대중교통)** - 경로 입력 → 영향도 체크 → 대체 경로 제시 |

#### Phase 2 (향후 확장)

| 기능 | 설명 |
|------|------|
| 파업 정보 자동 수집 | 크롤링 + LLM 파이프라인 |
| 위치 기반 자동 알림 | GPS 기반 맞춤 알림 |
| 택시 연동 | 택시 호출 + 요금 안내 |
| 카풀 매칭 | 같은 경로 유저 연결 |
| 지하철/철도 파업 확장 | 동일 프레임워크 확장 |

#### Phase 3 (B2B 확장)

| 기능 | 설명 |
|------|------|
| 버스 대절 | 회사 단위 전세버스 매칭 |
| HR 대시보드 | 직원 출퇴근 대응 관리 |
| 회사 공지 템플릿 | 자동 생성 |

### 4.5 유저 개인화 설정

유저가 직접 입력/설정하여 **맞춤형 알림과 대체 경로**를 받을 수 있는 정보.

#### 4.5.1 개인화 설정 항목

| 항목 | 설명 | 필수 여부 | Phase |
|------|------|-----------|-------|
| **출퇴근 경로** | 출발지/도착지 (주소 또는 역명) | 필수 | MVP |
| **관심 지역** | 서울 OO구, 경기 OO시 등 | 선택 | MVP |
| **관심 노선** | 평소 이용하는 버스 노선 | 선택 | MVP |
| **출퇴근 시간** | 평소 출발/도착 희망 시간 | 선택 | MVP |
| **알림 시간대** | 알림 받을 시간대 설정 | 선택 | Phase 2 |
| **알림 빈도** | 중요 알림만 / 모든 업데이트 | 선택 | Phase 2 |

#### 4.5.2 출퇴근 경로 설정

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 출발지 입력 | 주소 검색 또는 지도에서 선택 | P0 |

## 5. 파업 정보 수집 방식

### 5.1 정보 출처

파업 정보는 **신뢰도 순서**에 따라 수집한다.

#### 1차 출처 (최우선)

| 출처 | 설명 | 신뢰도 |
|------|------|--------|
| 서울시/경기도청 | 공식 홈페이지 보도자료, 공식 SNS | 최상 |
| 지자체 및 버스조합 | 공식 발표, 공지사항 | 상 |
| 중앙노동위원회 | 쟁의조정 신청 현황, 조정 결과 | 상 |

#### 2차 출처 (교차 검증용)

| 출처 | 설명 | 용도 |
|------|------|------|
| 주요 언론 | 연합뉴스, KBS, MBC, SBS 등 | 교차 검증 |
| 네이버 뉴스 | 검색 API 활용 | 모니터링 |

#### 3차 출처 (유저 제보)

| 출처 | 설명 | 용도 |
|------|------|------|
| 유저 제보 | 앱 내 제보 기능 | 실시간 현장 정보 |
| 커뮤니티 | 블라인드, 출퇴근 오픈채팅 등 | 보조 모니터링 |

**유저 제보 활용:**
- 공식 발표 전 현장 상황 빠른 파악
- 부분 파업/운행 재개 등 실시간 변동 감지
- 특정 노선/지역 상세 정보 수집

#### 출처 간 정보 상충 시 우선순위

```
1. 서울시/경기도청 공식 발표
2. 버스조합 공식 발표
3. 언론 보도 (복수 언론사 교차 확인)
4. 유저 제보 (복수 제보 교차 확인)
```

### 5.3 유저 제보 시스템

#### 5.3.1 제보 기능

| 기능 | 설명 | Phase |
|------|------|-------|
| 파업 제보 | "지금 버스 안 와요" 현장 제보 | MVP |
| 운행 재개 제보 | "버스 다시 다녀요" 정상화 제보 | MVP |
| 사진 첨부 | 현장 상황 사진 업로드 | Phase 2 |
| 위치 태그 | GPS 기반 제보 위치 자동 태그 | Phase 2 |

#### 5.3.2 제보 검증 프로세스

```
유저 제보 접수
      │
      ▼
동일 지역/노선 복수 제보 확인 (3건 이상)
      │
      ▼
운영팀 검토 또는 공식 출처 교차 확인
      │
      ▼
검증 완료 → 정보 반영 (유저 제보 태그 표시)
```

**검증 기준:**
- 단일 제보: 참고 정보로만 활용, 알림 발송 X
- 복수 제보 (3건+): 운영팀 검토 후 반영 검토
- 공식 출처 확인: 즉시 반영

#### 5.3.3 제보자 신뢰도 시스템 (Phase 2)

| 등급 | 조건 | 혜택 |
|------|------|------|
| 일반 | 신규 가입 | 기본 제보 |
| 신뢰 | 검증된 제보 5회 이상 | 제보 우선 검토 |
| 우수 | 검증된 제보 20회 이상 | 단독 제보도 검토 대상 |

### 5.4 정보 신뢰 기준

| 원칙 | 설명 |
|------|------|
| **1차 출처 우선** | 공식 기관 또는 언론을 1차 출처로 사용 |
| **상태 구분** | 추측/가능성 단계 정보는 상태값으로 명확히 구분 |
| **확정 정보만 알림** | 확정되지 않은 정보는 푸시 알림 대상에서 제외 |
| **유저 제보 표시** | 유저 제보 기반 정보는 "유저 제보" 태그 명시 |
| **루머 배제** | SNS 루머 및 미확인 정보는 절대 반영하지 않음 |
