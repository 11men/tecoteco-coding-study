## 1. 문제 정의

### 1.1 문제 상황

버스 파업이 발생하면 출퇴근자들은 다음과 같은 어려움을 겪는다:

- 파업 소식은 뉴스로 접하지만, 내가 타는 노선이 영향받는지 즉시 알기 어려움
- 대체 교통편을 직접 검색하고 비교해야 하는 시간 소모
- 출발 시간을 얼마나 앞당겨야 하는지 판단 어려움
- 파업 당일 새벽, 혼란 속에서 빠른 의사결정 필요

### 1.2 현재 대안의 한계

| 현재 대안 | 제공하는 것 | 부족한 점 |
|-----------|-------------|-----------|
| 뉴스/포털 | 파업 발생 사실 전달 | 내 노선 영향도, 대체 경로 없음 |
| 네이버/카카오 지도 | 일반 경로 탐색 | 파업 상황 미반영, 실시간 업데이트 없음 |
| 교통 정보 앱 | 실시간 도착 정보 | 파업 시 운행 중단 정보 부정확 |

### 1.3 핵심 인사이트

> *"파업 알림"은 이미 뉴스가 해준다. 아무도 안 해주는 건 "그래서 나는 어떻게 출근해?"이다.*

## 2. 타겟 유저

### 2.1 1차 타겟: 경기도 광역버스 출퇴근러

가장 타격이 큰 유저 그룹. 광역버스 의존도가 높고, 대체 교통편 접근성이 낮음.

### 2.2 페르소나

| 항목 | 내용 |
|------|------|
| 이름 | 김직장 (32세, 판교 IT기업 개발자) |
| 거주지 | 수원시 영통구 |
| 출근 루트 | 영통역 → 3100번 광역버스 → 판교역 (약 1시간) |
| Pain Point | 파업 뉴스 봤는데 3100번이 해당되는지 모르겠음. 지하철로 가면 1시간 40분인데, 몇 시에 나가야 하지? |
| 니즈 | 전날 밤에 알아서 새벽 지하철 탈 준비, 대체 경로와 소요시간 비교 |

### 2.3 확장 타겟

- 서울 시내버스 이용자: 대체 노선이 많아 영향도 확인만 필요
- 회사 총무/HR: 직원 출근 대응 공지 필요

---

## 3. 핵심 가치 제안

### 3.1 한 문장 정의

> **"파업 떴을 때, 30초 만에 나만의 대체 출근 루트를 알려준다"**

### 3.2 핵심 가치

1. **즉시성**: 파업 정보 확인부터 대체 경로 확인까지 30초 이내
2. **개인화**: 내 출퇴근 경로 기반 맞춤 정보
3. **실행 가능성**: 단순 정보가 아닌, 바로 행동할 수 있는 가이드

## 4. 기능 정의

### 4.1 MVP 기능 (2시간 내 구현)

| # | 기능 | 설명 | 우선순위 |
|---|------|------|----------|
| 1 | 출퇴근 경로 입력 | 출발지/도착지 입력 (주소 또는 지하철역) | P0 - 필수 |
| 2 | 파업 현황 표시 | 오늘 파업 여부, 영향 노선 목록 | P0 - 필수 |
| 3 | 내 노선 영향도 체크 | 입력한 경로가 파업 영향권인지 표시 | P0 - 필수 |
| 4 | 대체 경로 추천 | 지하철 기반 대체 경로 1~2개 제시 | P0 - 필수 |
| 5 | 소요시간 비교 | 평소 vs 대체 경로 시간 비교 | P1 - 권장 |

### 4.2 Phase 2 기능 (향후 확장)

- 푸시 알림: 파업 확정 시 전날 밤 알림
- 카풀 매칭: 같은 경로 유저 연결
- 회사 공지 템플릿: HR용 자동 생성
- 지하철/철도 파업 확장
- 날씨(폭설/태풍) 대응 통합

### 4.3 스코프 아웃 (안 할 것)

- 실시간 버스 위치 추적
- 택시 호출 연동
- 전국 버스 파업 대응 (서울/경기 집중)

## 5. 유저 플로우

### 5.1 메인 플로우

1. **[랜딩]** 오늘 파업 현황 한눈에 표시 (파업 있음/없음)
2. **[입력]** 출발지 → 도착지 입력
3. **[결과]** 내 경로 영향도 + 대체 경로 비교 표시
4. **[액션]** 권장 출발 시각, 지도 앱 연결

### 5.2 화면 구성 (MVP)

| 화면 | 구성 요소 |
|------|-----------|
| 메인 화면 | 파업 상태 배너, 출발지/도착지 입력 폼, 검색 버튼 |
| 결과 화면 | 영향도 표시, 평소 경로 카드, 대체 경로 카드, 시간 비교 차트 |

## 6. 정보 구조 및 데이터

### 6.1 필요 데이터

| 데이터 | 소스 | MVP 처리 방식 |
|--------|------|---------------|
| 파업 정보 | 뉴스 크롤링 + LLM 파싱 | 하드코딩 (데모용) |
| 영향 노선 목록 | 버스 조합 공지, 뉴스 | 하드코딩 (주요 광역버스) |
| 대체 경로 | 네이버/카카오 지도 API | API 연동 |
| 소요시간 | 지도 API | API 연동 |

### 6.2 데이터 파이프라인 (향후)

파업 뉴스 크롤링 → LLM으로 정보 추출 (날짜, 지역, 노선) → 노선 DB 매핑 → 유저 경로와 매칭 → 대체 경로 생성

## 7. 기술 검토

### 7.1 필요 API

| API | 용도 | 비고 |
|-----|------|------|
| 카카오 모빌리티 API | 대중교통 경로 탐색 | 무료 티어 존재 |
| 네이버 지도 API | 경로 탐색, 소요시간 | 일 25,000건 무료 |
| 공공데이터포털 | 버스 노선 정보 | 무료 |

### 7.2 플랫폼 전략

| 단계 | 플랫폼 | 이유 |
|------|--------|------|
| MVP | 웹앱 (PWA) | 빠른 프로토타이핑, 설치 없이 즉시 접근 가능 |
| 정식 출시 | 네이티브 앱 (iOS/Android) | 푸시 알림, 백그라운드 실행, 더 나은 UX |

> **MVP에서 웹앱을 선택한 이유**
> - 2시간 스프린트 내 구현 가능
> - 파업 상황에서 "링크 공유"로 빠른 확산 가능
> - 앱스토어 심사 없이 즉시 배포
> - 정식 버전에서 React Native 또는 Flutter로 앱 전환 예정

### 7.3 기술 스택 (MVP)

- 프론트엔드: React + Tailwind (빠른 프로토타이핑)
- 백엔드: 불필요 (클라이언트에서 직접 API 호출)
- 파업 데이터: JSON 하드코딩

### 7.4 제약사항

- 파업 정보 자동 수집 파이프라인 없음 (MVP에서는 수동 업데이트)
- 부분 파업(일부 노선만) 대응 어려움
- 실시간 파업 상황 변동 반영 불가

## 8. 경쟁 및 대안 분석

### 8.1 경쟁 환경

직접 경쟁자는 없음. 파업 대응에 특화된 서비스가 현재 시장에 존재하지 않음.

### 8.2 간접 대안

| 대안 | 우리의 차별점 | 전환 포인트 |
|------|---------------|-------------|
| 뉴스 검색 | 내 경로 영향도 즉시 확인 | "검색 시간 절약" |
| 지도 앱 | 파업 상황 반영 경로 비교 | "한 화면에서 비교" |
| 직접 문의 | 새벽에도 즉시 확인 가능 | "24시간 대응" |

## 9. GTM 전략

### 9.1 이벤트 드리븐 마케팅

파업은 이벤트성. 평소 유저 유입은 낮지만, 파업 발생 시 폭발적 수요.

### 9.2 채널 전략

- 파업 뉴스 댓글에 서비스 링크 시딩
- 출퇴근 관련 커뮤니티 (블라인드, 판교/강남 직장인 커뮤니티)
- 트위터/X에서 "버스 파업" 실시간 검색 대응
- 카카오톡 오픈채팅 (지역별 출퇴근 정보방)

### 9.3 바이럴 포인트

"파업 떴을 때 이거 쓰세요" → 유저가 직접 공유하는 구조 설계. 결과 화면에 공유 버튼 + 밈화 가능한 문구.

## 11. 성공 지표

### 11.1 MVP 성공 기준 (2시간 스프린트)

- 동작하는 프로토타입 완성
- 출발지/도착지 입력 → 대체 경로 표시 플로우 작동
- 팀원 모두가 "이거 파업 때 쓸 것 같다" 공감

### 11.2 실제 런칭 시 KPI

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| 파업 당일 DAU | 1,000+ | GA4 |
| 경로 검색 완료율 | 70%+ | 퍼널 분석 |
| 공유 전환율 | 10%+ | 공유 버튼 클릭 |
| 재방문율 (다음 파업) | 30%+ | 리텐션 분석 |
