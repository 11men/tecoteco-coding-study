## 1. 처음 접했을 때 전략

### 1.1 개별 PRD → 마스터 PRD 전략

각자 다른 관점에서 PRD를 작성한 후 하나로 통합하는 방식을 선택했습니다.

```
김OO의 PRD  ─┐
정OO의 PRD  ─┼──▶  PRD Master (통합본)
홍OO의 PRD  ─┘
```

**왜 이 전략을 선택했나?**
- 각자의 아이디어와 관점을 먼저 정리
- 토론 전에 개인의 생각을 명확히 문서화
- 통합 과정에서 더 나은 아이디어 도출
- 합의 과정이 자연스럽게 진행됨

### 1.2 "알림 앱"에서 "출퇴근 플랫폼"으로

처음에는 단순히 "파업 알림"을 주는 앱으로 시작했지만, 전략적으로 **정체성을 확장**했습니다.

| 초기 아이디어 | 확장된 정체성 |
|--------------|--------------|
| 파업 알림 앱 | **출퇴근 종합 플랫폼** |
| 정보 전달 | **문제 해결 (대안 제공)** |
| 무료 서비스 | **수익 창출 가능한 비즈니스** |

**핵심 인사이트:**
> "파업 알림은 뉴스가 이미 해준다. 우리가 해야 할 건 **'그래서 나는 어떻게 출근해?'**에 답하는 것이다."

## 2. 한정된 시간에 목표와 범위 산정

### 2.1 2가지 핵심 목표 설정

시간이 제한되어 있어서 **딱 2가지 핵심 목표**에 집중했습니다:

| 목표 | 핵심 질문 | 우선순위 |
|------|----------|----------|
| **목표 1: 신뢰도 높은 알림** | "파업이 언제, 어디서, 어떤 노선에?" | P0 |
| **목표 2: 대체 대안 제공** | "그래서 나는 어떻게 출근해?" | P0 |

### 2.2 MVP vs Phase 구분

모든 기능을 다 만들 수 없으므로 **MVP 범위를 명확히 정의**했습니다:

| 구분 | MVP (오늘 데모) | Phase 2 | Phase 3 |
|------|----------------|---------|---------|
| 알림 | 파업 상태 표시 | 푸시 알림 | 자동 수집 |
| 대안 1 | 대중교통 최적 경로 | - | - |
| 대안 2 | 택시 (UI만) | 택시 연동 | - |
| 대안 3 | 카풀 (UI만) | 카풀 매칭 | - |
| 대안 4 | 버스 대절 (UI만) | - | B2B 계약 |

### 2.3 수익 모델까지 고려

단순 프로토타입이 아닌 **비즈니스 관점**에서 접근했습니다:

```
┌─────────────────────────────────────────────────┐
│              수익 모델 구조                       │
├─────────────────────────────────────────────────┤
│  택시 호출     →  호출 수수료 (CPA)              │
│  카풀 매칭     →  매칭 수수료                    │
│  버스 대절     →  B2B 계약 수수료                │
│  광고          →  지도앱 연동 광고               │
└─────────────────────────────────────────────────┘
```

**GNB(하단 네비게이션)에 서비스를 노출**하여 파업이 아닐 때도 사용할 수 있는 **상시 서비스**로 확장했습니다.

## 3. AI 활용 방식

### 3.1 Claude를 활용한 워크플로우

```
[PRD 작성] → [PRD 통합/정리] → [프로토타입 코드 생성] → [UI 개선]
     ↑              ↑                    ↑                ↑
   사람 작성      Claude 지원         Claude 생성      Claude 개선
```

### 3.2 AI가 잘한 것

| 영역 | 활용 내용 |
|------|----------|
| **문서 통합** | 3개 PRD를 구조화된 마스터 PRD로 병합 |
| **코드 생성** | Next.js + Tailwind + lucide-react 프로토타입 |
| **UI 개선** | 석호님 코드 분석 후 개선점 반영 |
| **반복 작업** | 9개 페이지 일관된 스타일로 빠르게 생성 |

### 3.3 AI 활용 시 주의한 점

- **방향성은 사람이 결정**: "알림 앱 → 플랫폼" 전략은 팀 토론으로 결정
- **비즈니스 판단은 사람이**: 수익 모델, GNB 배치 등은 팀이 직접 기획
- **AI는 실행 도구로**: 코드 생성, 문서 정리 등 반복 작업에 활용

## 4. 한계점과 앞으로의 포인트

### 4.1 이번 세션의 한계

#### 1) 협업 병목 현상
```
┌─────────────────────────────────────────┐
│         3명이 1개의 Claude 공유          │
├─────────────────────────────────────────┤
│  A: Claude 사용 중  │  B: 대기  │  C: 대기  │
│  (작업 중)          │  (유휴)   │  (유휴)   │
└─────────────────────────────────────────┘
```

- **하나의 Claude 세션**을 3명이 돌아가며 사용
- 한 명이 작업할 때 **나머지 2명은 대기 상태**
- 역할 분업이 어려워 **병렬 작업 불가**

#### 2) 실제 기능 구현 부족
- 백엔드 없이 **프론트엔드 목업**만 완성
- 실제 데이터 연동, 푸시 알림 등 미구현
- 택시/카풀/버스대절은 **UI만 존재** (실제 동작 X)

### 4.2 개선 방향

| 한계 | 해결 방안 |
|------|----------|
| Claude 병목 | 역할별 Claude 세션 분리 (PRD용, 코드용, 디자인용) |
| 대기 시간 | 병렬 작업 가능한 태스크 분배 |
| 목업 한계 | 백엔드 API 구현 + 실제 데이터 연동 |

### 4.3 향후 발전 포인트

1. **실제 파업 데이터 연동**
   - 서울시/경기도 공식 API 연동
   - 뉴스 크롤링 + LLM 파이프라인

2. **서비스 확장**
   - 택시 앱 딥링크 연동
   - 카풀 매칭 알고리즘
   - B2B 버스 대절 계약 시스템

3. **PWA + 푸시 알림**
   - 웹앱을 PWA로 배포
   - FCM 기반 실시간 푸시 알림
